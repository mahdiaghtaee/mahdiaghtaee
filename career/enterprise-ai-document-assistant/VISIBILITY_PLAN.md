# Authentic Visibility Plan

The goal is to make the project discoverable because it is useful and well explained, not to manufacture activity.

## Repository Quality

- Keep the README accurate and concise.
- Maintain a runnable quick start.
- Keep limitations and security scope visible.
- Convert roadmap items into focused issues.
- Add tests and documentation with each meaningful feature.
- Avoid placeholder commits, artificial conversations, or exaggerated claims.

## Sharing

- Share the engineering case study, not only the repository link.
- Ask for feedback on one concrete design question, such as the pgvector abstraction.
- Post updates only when there is a meaningful technical change.
- Respond to questions with implementation details and trade-offs.
- Invite small, scoped contributions that can be reviewed properly.

## Suggested Milestones to Share

1. Initial local-first architecture and reproducible Docker setup.
2. Persistent pgvector semantic index.
3. Background document indexing with retries.
4. Authentication and document authorization.
5. OpenTelemetry traces and audit events.
6. A measured comparison of embedding or retrieval providers.

## What Not to Do

- Do not exchange stars or use fake accounts.
- Do not claim production readiness before security and persistence work is complete.
- Do not publish invented benchmarks, user counts, or performance results.
- Do not create low-value issues or commits only to make the repository appear active.
